### 3.3 — Health-check eviction termina a EC2 (sem orphans)

Quando um worker falha 3 health checks consecutivos, o `WorkerPool` invoca o callback registado pelo `AutoScaler`, que termina a EC2 — evitando instâncias órfãs.

```bash
# 1) Lança 1 worker, mata-o brutalmente (simula app crash sem instance crash)
WORKERS=( $(aws ec2 describe-instances --region eu-west-1 \
  --filters "Name=tag:Project,Values=NatureAtCloud" "Name=tag:Role,Values=worker" "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].InstanceId' --output text) )
[ ${#WORKERS[@]} -ge 2 ] || { echo "Preciso ≥2 workers; força scale-up antes"; exit 1; }
KILL="${WORKERS[-1]}"
echo "A matar $KILL"
# Em vez de terminate (que já desliga a VM), "freeze" o processo Java via SSH para simular crash
IP=$(aws ec2 describe-instances --instance-ids $KILL --region eu-west-1 \
      --query 'Reservations[0].Instances[0].PublicIpAddress' --output text)
ssh -i scripts/cnv-keypair.pem ec2-user@$IP "sudo systemctl stop cnv-worker.service"

# 2) Aguarda ~50s (3 falhas × 15s) e confirma nos logs do LB
sleep 55
LB_IP=$(aws ec2 describe-instances --region eu-west-1 \
  --filters "Name=tag:Project,Values=NatureAtCloud" "Name=tag:Role,Values=loadbalancer" "Name=instance-state-name,Values=running" \
  --query 'Reservations[0].Instances[0].PublicIpAddress' --output text)
ssh -i scripts/cnv-keypair.pem ec2-user@$LB_IP \
    "grep -E '(Health check FAILED|Removing unhealthy|terminada por health-check)' /opt/cnv/lb.log | tail"
# ✅ Esperado:
#   [WorkerPool] Health check FAILED (3/3): <ip>:8000 [i-XXXX]
#   [WorkerPool] Removing unhealthy worker: <ip>:8000 [i-XXXX]
#   [AutoScaler] EC2 i-XXXX terminada por health-check eviction.

# 3) Confirma na AWS que a EC2 foi mesmo terminada
aws ec2 describe-instances --instance-ids $KILL --region eu-west-1 \
    --query 'Reservations[].Instances[].State.Name' --output text
# ✅ "shutting-down" ou "terminated"
```

## Troubleshooting comum

**LB não responde em 8080 após 2 min:**
```bash
ssh -i scripts/cnv-keypair.pem "ec2-user@$(cat scripts/.state/lb-ip.txt)" \
    "cat /var/log/cnv-bootstrap.log; echo ---; cat /opt/cnv/lb.log 2>/dev/null | tail -50"
```
Causas frequentes: `iam:PassRole` em falta (re-corre `01-setup-iam.sh`), JAR não copiado (re-corre `05-launch-lb.sh`), porta 8080 fechada (verificar SG `cnv-lb-sg`).

**Worker arranca mas não responde:**
```bash
ssh -i scripts/cnv-keypair.pem "ec2-user@<worker-ip>" \
    "sudo systemctl status cnv-worker.service; sudo journalctl -u cnv-worker.service -n 50"
```

**`aws sts get-caller-identity` falha:**
- Credenciais AWS Academy expiraram (sessão dura ~3 h). Re-fazer `aws configure` ou exportar `AWS_SESSION_TOKEN`.

**`DynamoDB scan` retorna 0 itens:**
- IAM role do worker (`CNV-Worker-Role`) não tem permissão `dynamodb:PutItem`. Verifica policy attached.
- A região não bate certo entre worker e CLI (`AWS_REGION` vs `--region`). Tudo deve estar em `eu-west-1`.

**`02-setup-network.sh` falha com "Não consegui detectar IP":**
- Estás atrás de proxy/firewall que bloqueia `checkip.amazonaws.com`. Define manualmente:
  ```bash
  MY_IP=$(curl -s https://ifconfig.me)  # ou descobre noutro lado
  bash scripts/02-setup-network.sh
  ```
