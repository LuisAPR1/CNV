# FIX 04 — Contagem REAL de Basic Blocks (ControlFlow)

> **Data:** 2026-05-21
> **Ficheiro alterado:** `javassist/src/main/java/pt/ulisboa/tecnico/cnv/javassist/JavassistAgent.java`
> **Substitui a limitação documentada em** `00-project-status-and-roadmap_pt.md` §5.2 A

---

## Problema

A versão anterior do `JavassistAgent.instrumentMethod()` fazia:

```java
int codeLength = method.getMethodInfo().getCodeAttribute().getCodeLength();
long estimatedBlocks = Math.max(1, codeLength / 15);
method.insertBefore(
    "{ MetricRegistry.incrementBasicBlocks(" + estimatedBlocks + "L); }");
```

Ou seja:

- O número de "basic blocks" era uma **constante calculada em tempo de instrumentação** (uma fração do tamanho do bytecode do método).
- A chamada a `incrementBasicBlocks(N)` era injetada **uma única vez à entrada do método** (via `insertBefore`).
- Resultado: `basicBlockCount` somava sempre o mesmo `N` por invocação, **independentemente** dos parâmetros do pedido.

### Impacto concreto

`JuliaFractal.generate(w, h, iter)` contém todo o trabalho — `for y → for x → while` — num único método. Logo, por pedido:

| Pedido | Trabalho real | `basicBlockCount` antigo |
|---|---|---|
| `?w=100&h=100&iterations=50`     | ≤ 5 × 10⁵   | **K** (constante) |
| `?w=2000&h=2000&iterations=2000` | ≤ 8 × 10⁹   | **K** (mesma constante) |

O `ComplexityEstimator` ratio-based (`01.3_estimador_complexidade.md`) tenta aprender `basicBlocks ≈ k × feature(params)` a partir do histórico do DynamoDB. Com a métrica antiga, `basicBlocks` é praticamente constante → o `k` aprendido fica sem significado e o estimador acaba a depender do fallback heurístico.

O enunciado (`Project.txt` §2.4) exige métricas **dinâmicas** ("number of bytecode instructions or **basic blocks executed**"), justamente porque o wall-clock é pouco fiável sob contenção.

---

## Solução

Instrumentação ao nível do **basic block real**, usando a análise de fluxo de controlo do próprio Javassist (`javassist.bytecode.analysis.ControlFlow`).

### Algoritmo

Para cada método dos packages alvo (`fractals`, `grayscott`, `dna`):

1. Construir o CFG: `ControlFlow cf = new ControlFlow(ctClass, methodInfo)`.
2. Obter os basic blocks: `Block[] blocks = cf.basicBlocks()`.
3. Para cada `Block`, injetar à sua posição inicial o payload:
   ```
   lconst_1
   invokestatic MetricRegistry.incrementBasicBlocks(J)V
   ```
4. Inserir em **ordem decrescente de posição** (para que as posições reportadas pelo `ControlFlow` permaneçam válidas para os blocos anteriores).
5. Recalcular `maxStack` e reconstruir a `StackMapTable` (`rebuildStackMapIf6`).

### Porquê dynamic agora

Cada vez que o fluxo entra num basic block, o contador incrementa. No `JuliaFractal.generate`:

- O bloco do corpo do `while` é entrado **até `width × height × iterations` vezes**.
- O bloco do corpo do `for x` é entrado `width × height` vezes.
- O bloco do corpo do `for y` é entrado `height` vezes.

O `basicBlockCount` passa a ser **proporcional ao trabalho real**, que é exatamente o sinal que o LB precisa.

## Detalhes de implementação

### Ordem das três injeções por método

```
1.  instrumentBasicBlocks(method)         // posições do bytecode ORIGINAL
2.  method.insertBefore(incrementMethodCalls)
3.  if handler.handle: instrumentHandleMethod(method)   // startRequest/stopRequest
```

A inversão é essencial: `insertBefore` adiciona bytes ao prólogo do método e desloca todas as posições. Se o ControlFlow corresse depois, os `Block.position()` já estariam errados.

### Atualização de branch targets

`CodeIterator.insertAt(pos, bytes)` desloca automaticamente os offsets de todos os branches que apontam para posições ≥ `pos`. Não temos de fazer nada manualmente.

### Stack-map rebuild

Para class files Java 7+ a `StackMapTable` é obrigatória. Após qualquer modificação do bytecode, chama-se:

```java
codeAttribute.computeMaxStack();
methodInfo.rebuildStackMapIf6(classPool, classFile);
```

Sem isto o classloader rejeita a classe com `VerifyError`.

### Métodos pathological

O `ControlFlow` consegue falhar em métodos com CFGs estranhos (raro). Apanhamos a exceção e saltamos esse método — sem abortar a instrumentação do resto da classe.

---

## O que se mantém

- **`methodCallCount`** — útil para `Dna.runDnaSequenceMatcher` (que invoca `findSeed`/`extendRight` em loop).
- **`elapsedTimeMs`** — registado mas usado só como referência (o enunciado avisa que é ruidoso).
- **Assinaturas do `MetricRegistry`** — inalteradas, logo nada mais precisa de mudar (DynamoDB, ComplexityEstimator, LoadBalancer continuam a funcionar; só o **valor** que armazenam é que passa a ser útil).
