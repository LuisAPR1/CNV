# Cache Design — ComplexityEstimator (MSS Read-Through)

> Documentação do mecanismo de cache que protege o DynamoDB (MSS) de se
> tornar um gargalo no Load Balancer, conforme avisado no enunciado do
> projeto.

---

## Visão Geral

O `ComplexityEstimator` consulta o DynamoDB para obter histórico de
métricas e produzir estimativas de complexidade para pedidos novos.
Fazer uma query ao DynamoDB por cada pedido HTTP que chega ao LB seria
um **gargalo grave** — latência acrescida, custos de RCU, e risco de
throttling.

A solução é um **cache local em memória com TTL de 30 segundos**,
implementado como um `ConcurrentHashMap<String, CachedHistory>`.

```
┌──────────────┐     estimate()      ┌──────────────────────┐
│  LoadBalancer │ ──────────────────▶ │ ComplexityEstimator  │
│  (por request)│                     │                      │
└──────────────┘                      │  ┌────────────────┐  │
                                      │  │ cache (30s TTL)│  │
                                      │  │ requestType →  │  │
                                      │  │ CachedHistory  │  │
                                      │  └───────┬────────┘  │
                                      │          │ miss/exp  │
                                      │          ▼           │
                                      │  ┌────────────────┐  │
                                      │  │ DynamoDB Query │  │
                                      │  │ (últimos 50)   │  │
                                      │  └────────────────┘  │
                                      └──────────────────────┘
```

## Estrutura de Dados

### `ConcurrentHashMap<String, CachedHistory> cache`

- **Chave:** `requestType` (`"fractals"`, `"grayscott"`, `"dna"`)
- **Valor:** `CachedHistory` — wrapper com timestamp de criação
- **Thread-safe:** `ConcurrentHashMap` permite leituras concorrentes sem
  locks, essencial para o LB multi-threaded

### `CachedHistory`

```java
private static class CachedHistory {
    final List<HistoricalRecord> records;  // até 50 registos
    final long fetchedAt;                  // System.currentTimeMillis()

    boolean isExpired() {
        return System.currentTimeMillis() - fetchedAt > 30_000; // 30s
    }
}
```

### `HistoricalRecord`

```java
private static class HistoricalRecord {
    final Map<String, String> parameters;  // ex: {w=400, h=300, iterations=100}
    final long instructionCount;           // métrica ICount (CPU)
    final long allocatedBytes;             // métrica de alocação (RAM)
}
```

---

## Constantes de Configuração

| Constante | Valor | Justificação |
|---|---|---|
| `CACHE_TTL_MS` | 30.000 ms (30s) | Dados de métricas mudam lentamente; 30s é fresco o suficiente para estimativas e evita queries excessivas |
| `MAX_HISTORY_RECORDS` | 50 | Suficiente para média ratio-based estável; limita memória e RCUs |
| `DYNAMODB_RETRY_INTERVAL_MS` | 60.000 ms (60s) | Intervalo entre tentativas de reconexão se DynamoDB cair |

## Fluxo de Execução

### 1. Cache Hit (caso comum, ~95%+ dos pedidos)

```
estimate(requestType, params)
  └─ getHistory(requestType)
       └─ cache.get(requestType) → CachedHistory
            └─ !isExpired() → devolve records (0 queries DynamoDB)
```

**Latência:** ~0 ms (acesso a HashMap em memória)

### 2. Cache Miss / Expired

```
estimate(requestType, params)
  └─ getHistory(requestType)
       └─ cache.get(requestType) → null ou isExpired()
            └─ dynamoDB.query(...) → QueryRequest
                 ├─ KeyConditionExpression: "requestType = :rt"
                 ├─ ScanIndexForward: false (mais recentes primeiro)
                 └─ Limit: 50
            └─ cache.put(requestType, new CachedHistory(records))
            └─ devolve records
```

**Latência:** ~10-50 ms (query DynamoDB + parsing)

### 3. DynamoDB Indisponível

```
getHistory(requestType)
  └─ try { dynamoDB.query(...) }
     catch (Exception e) → log erro, devolve Collections.emptyList()
  └─ estimateFromHistory() → history.isEmpty() → return -1
  └─ fallback: estimateFromHeuristics() → estimativa param-based
```

**Latência:** ~0 ms (heurística local)

## Negative Caching

Se um `requestType` não tem registos no DynamoDB (ex: primeiro deploy,
tabela vazia), a lista vazia **também é cacheada**:

```java
List<HistoricalRecord> records = new ArrayList<>(); // vazia
CachedHistory newCache = new CachedHistory(records);
cache.put(requestType, newCache);  // ← cacheia lista vazia
return records;
```

Isto evita que **todos os pedidos** batam no DynamoDB enquanto a tabela
está vazia. Durante 30s, usam heurísticas. No ciclo seguinte, se já
houver dados, o cache é renovado.

---

## Invalidação

**Não há invalidação ativa.** O cache expira por TTL (30s).

Isto significa que métricas escritas por workers no DynamoDB demoram
até 30s a ser visíveis ao estimador do LB. Na prática, é irrelevante:

- O rácio `instructionCount / feature` converge rapidamente (poucas
  medições chegam para estabilizar a média)
- 30s de staleness não afetam decisões de routing porque a ordem de
  grandeza das estimativas não muda

---

## Cache Warming

**Não há aquecimento.** O cache começa frio.

No arranque do LB, os primeiros pedidos de cada tipo fazem query ao
DynamoDB. Com 3 workloads, são 3 queries "extra". Custo negligenciável
(~3 RCUs).

## Métricas de Cache (não implementadas)

Para o relatório, seria útil expor:

```
cacheHits:     N    // pedidos servidos do cache
cacheMisses:   N    // queries ao DynamoDB
cacheExpired:  N    // renovações por TTL
dynamodbErrors: N   // falhas de conexão
```

Implementação trivial (3 linhas) se necessário para evidência
quantitativa.

---

## Porquê Esta Abordagem e Não Outra?

| Alternativa | Problema |
|---|---|
| Sem cache (query por request) | DynamoDB seria gargalo: latência + RCUs + throttling |
| TTL mais curto (5s) | Queries demasiado frequentes sem ganho de frescura |
| TTL mais longo (5 min) | Dados stale; novos workloads sem histórico demoram a beneficiar |
| `Collections.synchronizedMap` | Pior performance que `ConcurrentHashMap` em leituras concorrentes |
| Guava Cache / Caffeine | Overkill — não precisamos de weight-based eviction nem stats |
| Cache distribuído (Redis/ElastiCache) | Complexidade desnecessária; LB é single-instance |
