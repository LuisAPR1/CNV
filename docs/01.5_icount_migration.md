## Para que serve este documento

Decidimos qual é o conjunto final de métricas usadas pelo sistema, justificámos a escolha à luz do enunciado e do FAQ, e migrámos o código todo de **basic-block count** para **instruction count (ICount)**. Este doc é referência para o relatório.

---

## Contexto: porque mudar do BB count para ICount

A FIX 04 (`01.4_basicblock_counting_real.md`) já tinha corrigido o problema fundamental: a métrica passou a ser **dinâmica** (escala com o trabalho real), em vez de constante por método. Mas continuava a contar basic blocks, com `+1` por cada entrada num bloco. Isto é defensável mas tem uma fragilidade:

> Um bloco com 50 instruções e um bloco com 3 instruções contavam **igual** (1 cada). A métrica perdia precisão sempre que os blocos do código tinham tamanhos heterogéneos.

ICount resolve isto: `+N` por entrada em cada bloco, onde `N` é o número de instruções bytecode desse bloco. **Mesma frequência de chamadas estáticas em runtime, mais informação.**

E o FAQ (`faq.txt:184`) é explícito:
> "Hint: use ICount as a starting point for your Javassist tool"

ICount é o standard recomendado. Não há razão para não migrar.

## A nossa proposta de métricas — três níveis

| Nível | Métrica | Para que serve | Onde é usada |
|---|---|---|---|
| **Primária** | `instructionCount` | Decisões de routing (LB) e scaling (AS). Estimativa de complexidade. | `ComplexityEstimator`, `WorkerPool.estimatedWork`, `AutoScaler.avgEstimatedWork` |
| **Secundária** | `methodCallCount` | Cross-check / diagnóstico. Útil sobretudo no DNA. | Guardada no DynamoDB, exposta nos logs `[Metrics]` |
| **Validação** | `elapsedTimeMs` | Verificar correlação entre estimativa e realidade. **Não usada para decisões** (o enunciado avisa que é ruidoso). | Diagnóstico no relatório, evidência empírica |

### Justificação de cada nível

**`instructionCount` como primária:**
- Standard CNV (FAQ menciona explicitamente).
- Dinâmica: escala com execução real (cada iteração de loop entra no bloco do corpo, contribuindo as suas N instruções).
- Robusta a estilo de código: refactorar `image.setRGB(x,y,c)` para `setR/setG/setB` não muda o número de instruções dentro do código de workload.
- Resolve o problema "100 calls caras vs 10000 calls baratas": uma chamada cara entra em mais blocos com mais instruções dentro → contagem reflete custo real.

**`methodCallCount` como secundária:**
- Útil em DNA, onde `findSeed`/`extendRight` correm em loop dentro do package.
- Cross-check: se o ratio `methodCallCount / instructionCount` muda muito entre pedidos do mesmo tipo, sinal de alarme (provavelmente o ratio de chamadas JDK varia).
- Manter custa zero: já estava a ser recolhida.

**`elapsedTimeMs` como validação:**
- Enunciado §2.4: usar wall-clock para decisões é ruidoso (workers competem por CPU).
- Mas é útil para **validar** o modelo: se ICount estimado predisser bem o tempo real, sabemos que a instrumentação está consistente.
- Para o relatório, dá evidência empírica (gráficos `instructionCount` vs `elapsedTimeMs`).

## Diferença concreta na instrumentação

### Antes (BB count, FIX 04):

```java
for (Block b : sorted) {
    Bytecode bc = new Bytecode(...);
    bc.addLconst(1L);                                          // ← +1 por bloco
    bc.addInvokestatic(METRIC_REGISTRY_INTERNAL,
            "incrementBasicBlocks", "(J)V");
    ci.insertAt(b.position(), bc.get());
}
```

### Depois (ICount, FIX 05):

```java
// Phase 1: count instructions per block (using ORIGINAL bytecode).
Map<Block, Integer> instructionsPerBlock = new HashMap<>();
for (Block b : blocks) {
    int n = countInstructionsInBlock(codeAttribute, b.position(), b.length());
    instructionsPerBlock.put(b, n);
}

// Phase 2: inject in descending order.
for (Block b : sorted) {
    int n = instructionsPerBlock.get(b);
    Bytecode bc = new Bytecode(...);
    bc.addLconst((long) n);                                    // ← +N por bloco
    bc.addInvokestatic(METRIC_REGISTRY_INTERNAL,
            "incrementInstructions", "(J)V");
    ci.insertAt(b.position(), bc.get());
}
```

### O helper `countInstructionsInBlock`

Para contar instruções num intervalo `[start, start+length)` do bytecode:

```java
private int countInstructionsInBlock(CodeAttribute ca, int start, int length) {
    int end = start + length;
    int count = 0;
    CodeIterator ci = ca.iterator();
    ci.move(start);
    while (ci.hasNext()) {
        int pos = ci.next();
        if (pos >= end) break;
        count++;
    }
    return Math.max(1, count);
}
```

Importante: **a contagem corre na fase 1, antes de qualquer injeção**. Se corresse depois, o bytecode já estaria deslocado e os offsets dos blocos seguintes seriam inválidos.

## Mudanças noutros ficheiros

### `MetricRegistry.java`

- Renomeado: `basicBlockCount` → `instructionCount` em `CompletedRequest` e `RequestMetrics`.
- Renomeado: `incrementBasicBlocks(long)` → `incrementInstructions(long)`.
- Renomeado: `getBasicBlockCount()` → `getInstructionCount()`.
- `toMap()` agora escreve a key `"instructionCount"`.
- `toString()` mostra `instructions=N` em vez de `basicblocks=N`.

### `MetricsStorageService.java`

- O atributo escrito no DynamoDB passou de `basicBlockCount` para `instructionCount`.
- Comentário explicativo adicionado sobre os 3 níveis de métricas.
- **Nota:** registos antigos no DynamoDB com `basicBlockCount` (escrita antes de hoje) **continuam lá** mas são **ignorados** pelo `ComplexityEstimator`.

### `ComplexityEstimator.java`

- `HistoricalRecord.basicBlockCount` → `instructionCount`.
- `Estimate.estimatedBasicBlocks` → `estimatedCost` (mais neutro semanticamente).
- `getEstimatedBasicBlocks()` → `getEstimatedCost()`.
- O leitor do DynamoDB lê `"instructionCount"`. Registos antigos com apenas `basicBlockCount` ficam com `instructionCount = 0` e são filtrados pelo modelo ratio-based.
- Heurísticas de fallback **recalibradas** para a escala de instruções:
  - Fractals: `w * h * iter * 12` (era `*10`)
  - GrayScott: `size² * maxIter * 30` (era `*5`)
  - DNA: `(seqLen1 * seqLen2 / minLen) * 10`

### `LoadBalancer.java`

- Substituída a chamada `estimate.getEstimatedBasicBlocks()` por `estimate.getEstimatedCost()`.

## Migração do DynamoDB — IMPORTANTE para deployment

A tabela `cnv-metrics` na AWS pode conter ~467 registos antigos com:
- `basicBlockCount` (campo antigo) — vai ser ignorado.
- Sem `instructionCount` (não existe no schema antigo).

Para o ratio-based aprender bem, é preciso ter pelo menos alguns pedidos com `instructionCount > 0`. Há duas opções:

**Opção A — Fresh start (recomendada):**
```bash
aws dynamodb delete-table --table-name cnv-metrics --region eu-west-1
# A tabela é re-criada automaticamente pelo MetricsStorageService no primeiro pedido.
```

**Opção B — Manter histórico antigo:**
- O sistema funciona à mesma — registos antigos são ignorados pelo estimador.
- Os primeiros N pedidos novos preenchem o histórico válido.
- Nenhuma ação necessária.

Como os dados antigos estão poluídos pela métrica errada original (BB constante, FIX 04 ainda não tinha corrido em produção), **opção A é claramente preferível**.
