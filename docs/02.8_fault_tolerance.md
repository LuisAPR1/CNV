## Objetivo

O enunciado (`Project.txt` §2.1) pede explicitamente:

> "If workers fail, the LB should handle this and hide failures from users."

O sistema precisa de **tolerância a falhas** ao nível dos workers (EC2 podem crashar, ficar lentos, ou ter problemas de rede). O LB é o ponto central para garantir que o cliente não vê erros sempre que possível.

## Mecanismos Principais de Fault Tolerance

### 1. Retry com Exclusão de Workers Falhados (Load Balancer)

**Local:** `LoadBalancer.ForwardHandler.handle()` (linhas ~134-184)

- O LB tenta encaminhar o pedido para até **3 workers diferentes** (ou menos, consoante o tamanho do pool).
- Quando um worker falha (exceção, timeout, erro de rede, 5xx), o LB:
  1. Regista a falha.
  2. Adiciona o worker ao conjunto `triedWorkers`.
  3. Pede ao `WorkerPool.selectForRequest(estimatedCost, triedWorkers)` um worker **diferente**.
- Usa a mesma estratégia híbrida de packing/spreading em cada tentativa.

Isto evita que um único worker morto "mate" o pedido.

### 2. Lambda como Fallback de Fault Tolerance

**Local:** `LoadBalancer.java` (comentário explícito na linha 188)

```java
// Lambda fallback: if EC2 retries exhausted and request is Lambda-eligible,
// try Lambda as a last resort (fault tolerance).
if (estimatedCostSeconds <= LAMBDA_MAX_SECONDS && lambdaInvoker.isAvailable()) {
    System.out.println("[LoadBalancer] EC2 retries exhausted — falling back to Lambda.");
    ...
}
```

Existem dois caminhos Lambda relacionados com resiliência:

- **Fast-path** (proativo): se todos os workers estiverem >80% ocupados + pedido elegível (≤5s estimado) → envia direto para Lambda (evita sobrecarregar ainda mais os workers).
- **Fallback reativo**: depois de esgotar todos os retries EC2 → usa Lambda como último recurso.

Isto dá uma **via alternativa de execução** quando a infraestrutura de VMs está em problemas.

### 3. Health Checks Periódicos + Remoção Automática

**Local:** `WorkerPool.java`

- Scheduler a cada **15 segundos** (`HEALTH_CHECK_INTERVAL_SECONDS`).
- Cada worker é pingado no endpoint `/` (RootHandler).
- Contador de falhas consecutivas por worker (`consecutiveFailures`).
- Ao atingir **3 falhas consecutivas** (`FAILURES_BEFORE_REMOVAL = 3`):
  - O worker é removido do pool (`removeWorker`).
  - É disparado o callback `onUnhealthyEviction` (se registado).
  - Existe lógica de recuperação: se o worker voltar a responder, o contador é limpo e aparece `"Health recovered"`.

### 4. Terminação de Instâncias EC2 em Caso de Falha (Eviction)

**Local:** `AutoScaler.java` + registo em `AutoScaler` constructor

```java
workerPool.setOnUnhealthyEviction(this::handleUnhealthyEviction);

private void handleUnhealthyEviction(WorkerPool.Worker w) {
    if (!awsMode || w.getInstanceId() == null) return;
    ec2.terminateInstances(...);
    System.out.println("[AutoScaler] EC2 " + ... + " terminada por health-check eviction.");
}
```

**Porque é importante?**  
Antes deste mecanismo, o WorkerPool removia o worker do pool mas a instância EC2 ficava a correr (órfã). Isto era um bug de custo + limpeza.

O callback garante que quando o LB detecta que um worker "morreu", a infraestrutura também é limpa.

### 5. Outros Mecanismos de Resiliência

- **Safe scale-down com defer** (`AutoScaler.scaleDown`): se ainda houver pedidos ativos após 30s de drenagem, o worker é **re-adicionado** ao pool em vez de ser terminado. Evita perda de pedidos em curso.
- **Retry no ComplexityEstimator**: tenta reconectar ao DynamoDB a cada 60s (`retryIfNeeded`). Enquanto não consegue, usa heurísticas (degradação graciosa).
- **Degradação geral**:
  - Sem DynamoDB → só heurísticas.
  - Sem credenciais AWS → modo local (só logs).
  - Sem Lambda disponível → só EC2.
- **Idempotência** em várias operações (addWorker, criação de tabela DynamoDB, deploy de Lambdas).

## Fluxo Típico Quando um Worker Falha

1. Worker crasha ou fica irresponível.
2. Health check (15s) falha → contador sobe.
3. Ao 3º falha consecutiva:
   - Worker removido do pool do LB (novos pedidos já não vão para ele).
   - AutoScaler recebe callback → termina a EC2.
4. Pedidos que estavam a ser servidos por esse worker:
   - O LB que estava a tentar reenvia para outro worker (retry).
   - Se todos falharem → fallback para Lambda (se elegível).
5. AutoScaler pode fazer scale-up se a carga média subir.

O cliente idealmente só vê o pedido atrasado (retry), não um erro 5xx.

## Decisão de Routing em Capacidade Cheia: Lambda para Pedidos Pequenos (em vez de nova VM)

Este é um ponto de design intencional e merece explicação clara, especialmente face a feedback do tipo:

> "Talvez faça sentido reencaminhar para outra vm ativa antes do lambda, ou até mesmo abrir uma vm ao invés de mandar para um lambda."

**A nossa posição (confirmada):**  
Quando as VMs estão cheias (todos os workers acima de ~80% da capacidade estimada), **pedidos pequenos** (estimativa de wall-clock ≤ 5 segundos) devem ir para Lambda em vez de lançar uma nova VM.

### Razões para esta escolha

1. **Cold start da VM é proibitivo para trabalho pequeno**
   - Uma t3.micro nova demora tipicamente 45–60 segundos até estar a servir (boot + systemd + Java).
   - Um pedido pequeno (ex: fractals 200×200 ou DNA curto) demora < 1–2 segundos.
   - Lançar uma VM nova para servir 1 segundo de trabalho é extremamente ineficiente em latência e custo.

2. **Lambda como válvula de escape rápida (não como caminho primário)**
   - O fast-path só dispara quando **todas** as VMs estão saturadas.
   - O AutoScaler continua a correr em paralelo (a cada 5s). Se a carga for sustentada, ele vai lançar novas VMs de forma normal (baseado em avgWorkSeconds > 2.5s).
   - O Lambda serve apenas para absorver picos triviais durante o tempo que a nova VM demora a arrancar.

3. **Separação clara por tamanho de pedido (alinhado com o enunciado)**
   - Pedidos grandes (estimatedCostSeconds > 5s) **nunca** vão para Lambda (nem no fast-path nem no fallback). Forçam o uso de EC2 e eventualmente scale-up.
   - Só pedidos pequenos são elegíveis para Lambda quando o pool está cheio.

4. **Alinhamento com os objetivos do projeto**
   - O enunciado pede explicitamente balancear EC2 (mais barato por pedido, arranque lento) vs Lambda (mais caro por pedido, arranque rápido).
   - Para trabalho muito curto, o custo por segundo em Lambda é irrelevante comparado com o custo de uma VM idle durante 1 hora.

### Fluxo resultante (pedido pequeno com pool cheio)

- Estimativa ≤ 5s + todos os workers > 80% → **Lambda direto** (fast-path).
- Se o Lambda também falhar → cai no caminho normal de retries (que entretanto pode ter scale-up a decorrer).
- Se a carga se mantiver alta → o AutoScaler lança VMs novas (scale-up normal).

Isto garante que pedidos pequenos não "pagam" o cold start de uma VM nova, enquanto pedidos pesados e carga sustentada continuam a forçar escala de VMs.

## Notas para o Relatório Final

**O que mencionar (sugestão de estrutura):**

1. **Motivação** — Citação direta do enunciado ("If workers fail, the LB should handle this...").
2. **Camadas de defesa** (em ordem de proximidade com o cliente):
   - Retry + exclusão de workers falhados (nível de pedido) — reencaminhamento para VMs ativas.
   - Lambda como último recurso / válvula de escape (fault tolerance + alívio de capacidade para trabalho pequeno).
   - Monitorização ativa (health checks) + limpeza automática de recursos + scale-up reativo.
3. **Decisões de design chave (e como responder à citação do professor)**:
   - Usar health checks simples em `/`.
   - 3 falhas antes de remover (evita remoções por flutuações temporárias).
   - Callback de eviction para evitar órfãos.
   - **Lambda fast-path apenas para pedidos pequenos quando pool saturado** — justificado por cold start de VM (~60s) vs tempo de execução do pedido. Não é "enviar tudo para Lambda"; é alívio temporário para trabalho trivial enquanto o AutoScaler reage.
   - Proteção forte: pedidos grandes **nunca** vão para Lambda.
4. **Evidência empírica**:
   - Script de resiliência + logs onde se vê "Health check FAILED (3/3)" → remoção + LB continua a servir.
   - Calibração de throughput t3.micro (bench-t3micro-throughput.csv) e thresholds em segundos de wall-clock.
   - Logs de produção mostrando fast-path a ser usado apenas em cenários de burst trivial.
5. **Limitações conhecidas** (sê honesto):
   - Não há circuit-breaker mais avançado.
   - O retry é síncrono (pode aumentar latência do cliente).
   - Lambda tem cold start (usado só como alívio temporário).
   - Decisão de preferir Lambda sobre nova VM para trabalho <5s é uma escolha de compromisso (pode ser discutida).

**Frases úteis para o relatório/oral (para responder diretamente à citação):**

- "Temos estratégia clara tanto para falha de pedido como para falha de VM: retries com exclusão de workers falhados + health checks com terminação automática da instância."
- "Para pedidos pequenos quando as VMs estão cheias, preferimos Lambda em vez de lançar imediatamente uma nova VM. Um pedido de 1 segundo não justifica 60 segundos de cold start de uma t3.micro. O AutoScaler continua a monitorizar e lança VMs se a carga for sustentada."
- "Pedidos grandes nunca vão para Lambda — o threshold de 5 segundos garante isso. A estratégia é exatamente a que o projeto pede: balancear o arranque lento mas barato do EC2 com o arranque rápido da Lambda."
- "O fast-path para Lambda é uma válvula de escape, não o caminho primário. Quando há capacidade nas VMs existentes, o sistema reencaminha sempre primeiro para elas (packing + retries)."
