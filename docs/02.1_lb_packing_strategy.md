## Para que serve este documento

## O problema de routing

Quando chega um pedido `/fractals?w=2000&h=2000&iterations=2000` ao LB, o LB tem de decidir **para qual dos N workers** o despacha. Existem três políticas clássicas:

### 1. Round-robin
Atribui pedidos por turnos: pedido 1 → W1, pedido 2 → W2, pedido 3 → W3, pedido 4 → W1...

- ❌ Ignora o trabalho real. Manda pedidos pesados para qualquer worker.
- ❌ Pode meter pedido pesado num worker que já tem outro pedido pesado.

### 2. Spreading puro (least-loaded)
Manda sempre para o worker com **menos** carga acumulada.

- ✅ Distribui carga uniformemente.
- ✅ Menor latência média (workers paralelizam ao máximo).
- ❌ **Mantém todos os workers ligeiramente carregados** → nunca há candidato natural a scale-down → custo $$$ alto.
- ❌ Não maximiza throughput por dólar.

### 3. Packing puro (most-loaded com fit)
Manda para o worker mais cheio que ainda tenha capacidade.

- ✅ Consolida trabalho → workers idle aparecem rapidamente → scale-down agressivo → custo baixo.
- ❌ Latência de cada pedido individual aumenta (filas dentro do worker).
- ❌ Risco de saturação se a estimativa de capacidade for errada.

---

## A nossa escolha: híbrido (packing dentro de teto, spreading como fallback)

```
selectForRequest(requestCost, excluded):
  bestPack   = null  // most-loaded worker still under cap
  bestSpread = null  // least-loaded worker (any cap)

  for each worker w not in excluded:
    load      = w.estimatedWork
    projected = load + requestCost

    # spreading fallback candidate (always tracked)
    if bestSpread is null OR load < bestSpread.load:
      bestSpread = w

    # packing candidate (only if it fits under the cap)
    if projected ≤ MAX_CAPACITY:
      if bestPack is null OR load > bestPack.load:
        bestPack = w

  return bestPack if bestPack is not null else bestSpread
```

### Como se comporta na prática

Cenário com 3 workers e `MAX_CAPACITY = 4×10¹¹`:

| Worker | `estimatedWork` actual | Projected (após +1×10¹¹) | Decisão |
|---|---|---|---|
| W1 | 3×10¹¹ | 4×10¹¹ ✅ fit | **PACK** (mais carregado) |
| W2 | 1×10¹¹ | 2×10¹¹ ✅ fit | (descartado: menos carregado que W1) |
| W3 | 0 (idle) | 1×10¹¹ ✅ fit | (descartado: ainda menos carregado) |

→ Pedido vai para **W1**. W3 fica **idle** e é candidato a scale-down no próximo ciclo do AutoScaler.

Cenário em que packing falha (todos os workers saturados):

| Worker | `estimatedWork` actual | Projected (+1×10¹¹) | Decisão |
|---|---|---|---|
| W1 | 4×10¹¹ | 5×10¹¹ ❌ over cap | (não pack) |
| W2 | 4×10¹¹ | 5×10¹¹ ❌ over cap | (não pack) |
| W3 | 3.5×10¹¹ | 4.5×10¹¹ ❌ over cap | (não pack) |

→ `bestPack = null`. Cai no **bestSpread = W3** (menos carregado entre os saturados). O pedido é servido (fila no W3) e o AutoScaler vê o pico no próximo ciclo de 5s e dispara scale-up.

## Porque é a melhor escolha para este projeto

O enunciado §5.1 pede minimização de **custo**:

> "Total cost = (EC2 hours × cost) + (Lambda invocations × cost)"

**Spreading puro** mantém todos os workers ligeiramente ocupados → cluster nunca consolida → mais EC2 hours.
**Packing puro** maximiza consolidação mas pode causar latência má se o cap estiver mal calibrado.
**Híbrido** dá:
- ✅ Consolidação na maior parte do tempo (packing) → menos EC2 hours.
- ✅ Safety net (spreading) quando todos os workers estão cheios → não degrada catastroficamente.
- ✅ Cost-aware por construção: cada decisão olha para `requestCost` actual, não só para o estado dos workers.

## Calibração do `MAX_CAPACITY`

Constante: `WorkerPool.DEFAULT_MAX_CAPACITY = 400_000_000_000L` (4×10¹¹ instruções).

Justificação dos números (em instruções bytecode):

| Pedido representativo | `estimatedCost` aproximado |
|---|---|
| Fractals 2000×2000×2000 (stress test) | ~10¹¹ |
| Fractals 800×800×500 (médio) | ~3×10⁹ |
| GrayScott 512×10000 | ~10¹¹ |
| DNA HBB×HBB minLen=5 | ~10⁵ |

Cap em 4×10¹¹ permite empilhar até 4 pedidos do tipo "stress test" num worker antes de cair no fallback de spreading. Para pedidos médios, há muita margem.

**Importante:** este valor é uma **calibração inicial baseada em estimativas teóricas**. Vai ser re-afinado depois da fase de validação empírica com a nova métrica ICount. ## Integração com o resto do sistema

### Ligação ao `ComplexityEstimator`

O `LoadBalancer.ForwardHandler.handle()` faz:

```java
ComplexityEstimator.Estimate estimate =
    complexityEstimator.estimate(requestType, params);
long estimatedCost = estimate.getEstimatedCost();
WorkerPool.Worker worker =
    workerPool.selectForRequest(estimatedCost, triedWorkers);
```

A qualidade do routing depende **directamente** da qualidade da estimativa. Daí a importância da migração para ICount (`01.5_icount_migration.md`) e da calibração das features no estimador.

### Ligação ao `AutoScaler`

O `AutoScaler.checkAndScale()` continua a usar `avgEstimatedWork` por worker. Com packing:
- Workers idle (load = 0) aparecem rapidamente quando a carga total baixa.
- O critério `avgEstimatedWork < THRESHOLD/4` dispara scale-down.
- Combinado com o **scale-down seguro** (`02.2_safe_scale_down.md`), o cluster consolida sem matar pedidos.

### Retry em caso de falha

O loop de retry no `LoadBalancer.ForwardHandler` passa o set `triedWorkers` ao `selectForRequest`. Workers que falharam o pedido (timeout, 5xx) são excluídos das tentativas subsequentes. **A política híbrida aplica-se em cada tentativa**, não há fallback diferente para retries.

---

## O que substituiu

A versão anterior usava `selectLeastLoadedExcluding(triedWorkers)` — spreading puro. Esta foi mantida como `selectLeastLoaded()` para uso interno, mas a rota de produção (`LoadBalancer.ForwardHandler`) usa agora `selectForRequest(cost, excluded)`.

### Diff conceptual

| Aspeto | Antes (spreading) | Depois (híbrido) |
|---|---|---|
| Política | Always least-loaded | Pack-then-spread |
| Cost-aware | Não (só olha `estimatedWork`) | Sim (compara `load + cost` ao cap) |
| Workers idle preserved | Não | Sim (deixa idle se há outro com fit) |
| Cap configurável | N/A | `DEFAULT_MAX_CAPACITY` |
| Saturação handling | Pedido vai à mesma, sem sinal | Pedido vai à mesma, sinal claro nos logs |

## Como testar

```bash
# Local: arrancar 3 workers em portas diferentes
java -javaagent:javassist/target/javassist-agent-1.0.0-SNAPSHOT-jar-with-dependencies.jar \
     -cp webserver/target/webserver-1.0.0-SNAPSHOT-jar-with-dependencies.jar \
     pt.ulisboa.tecnico.cnv.webserver.WebServer 8001 &
java -javaagent:... ... 8002 &
java -javaagent:... ... 8003 &

# Arrancar LB
java -cp loadbalancer/target/loadbalancer-1.0.0-SNAPSHOT-jar-with-dependencies.jar \
     pt.ulisboa.tecnico.cnv.loadbalancer.LoadBalancer 8080 \
     localhost:8001 localhost:8002 localhost:8003

# Disparar pedidos com magnitudes diferentes em paralelo
curl "http://localhost:8080/fractals?w=2000&h=2000&iterations=2000" &  # pesado
curl "http://localhost:8080/fractals?w=200&h=200&iterations=50"        # leve
```

Esperado nos logs do LB:
- Primeiro pedido (pesado): vai para qualquer worker (todos a 0).
- Segundo pedido (leve): vai para o **mesmo worker** que recebeu o pesado (packing — está mais carregado mas ainda cabe).

Se fosse spreading puro, o segundo iria para um worker idle.
