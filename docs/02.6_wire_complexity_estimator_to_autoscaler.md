# FIX 02.6 — Ligar ComplexityEstimator ao AutoScaler

> **Data:** 2026-05-21
> **Ficheiros alterados:**
> - `loadbalancer/.../WorkerPool.java` — `Worker` com `estimatedWork`, seleção baseada em custo
> - `loadbalancer/.../LoadBalancer.java` — `ForwardHandler` passa estimativas aos workers
> - `loadbalancer/.../AutoScaler.java` — decisões de scaling baseadas em `estimatedWork`

---

## Problema

Antes desta alteração, o `ComplexityEstimator` calculava o custo estimado de cada pedido (em basic blocks) mas o resultado era **apenas logado** — nunca influenciava as decisões do sistema:

1. **Routing cego:** O `WorkerPool.selectLeastLoaded()` escolhia o worker com menos `activeRequests`, ignorando que um worker com 1 pedido `2000×2000×2000` está muito mais carregado do que outro com 3 pedidos `100×100×50`.

2. **AutoScaler cego:** O `checkAndScale()` usava `avgLoad = totalActive / numWorkers`. Com o threshold de 1.0, bastavam 2 pedidos simultâneos (mesmo que triviais) para disparar scale-up — desperdiçando dinheiro. Pior: um único pedido pesadíssimo NUNCA disparava scale-up porque `totalActive` era só 1.

3. **Scale-down cego:** A seleção do worker a terminar usava `activeRequests`, podendo escolher um worker com 1 pedido pesado em vez de um com 2 pedidos leves.

## Solução

### Nova métrica: `estimatedWork` por worker

Cada `Worker` passou a acumular a soma dos `estimatedBasicBlocks` de todos os pedidos ativos que lhe foram atribuídos. Esta métrica reflete o **trabalho real pendente**, não apenas a contagem de pedidos.

### 1. WorkerPool.Worker — tracking de trabalho estimado

```java
// NOVO: campo na classe Worker
private final AtomicLong estimatedWork = new AtomicLong(0);

public long getEstimatedWork() { return estimatedWork.get(); }
public void addEstimatedWork(long work) { estimatedWork.addAndGet(work); }
public void removeEstimatedWork(long work) { estimatedWork.addAndGet(-work); }
```

O `toString()` foi atualizado para mostrar também `estWork=`.

### 2. WorkerPool — seleção baseada em custo

`selectLeastLoaded()` e `selectLeastLoadedExcluding()` passaram a usar `estimatedWork` como métrica primária:

```java
// Antes: w.getActiveRequests() < best.getActiveRequests()
// Agora:
if (w.getEstimatedWork() < best.getEstimatedWork()) {
    best = w;
} else if (w.getEstimatedWork() == best.getEstimatedWork()
        && w.getActiveRequests() < best.getActiveRequests()) {
    best = w;  // fallback: desempate por contagem
}
```

### 3. LoadBalancer.ForwardHandler — injeção da estimativa

O handler agora passa o custo estimado ao worker selecionado **antes** de encaminhar e remove-o **depois** (no `finally`):

```java
ComplexityEstimator.Estimate estimate = complexityEstimator.estimate(requestType, params);
long estimatedCost = estimate.getEstimatedBasicBlocks();

// ...
worker.incrementActive();
worker.addEstimatedWork(estimatedCost);   // NOVO
try {
    // ... forward request ...
} finally {
    worker.removeEstimatedWork(estimatedCost);  // NOVO
    worker.decrementActive();
}
```

### 4. AutoScaler — decisões baseadas em trabalho estimado

As constantes `SCALE_UP_THRESHOLD` (1.0) e `SCALE_DOWN_THRESHOLD` (0.25) foram substituídas por:

```java
/**
 * Limiar de trabalho estimado por worker (em basic blocks).
 * ~1 bilião BBs ≈ um pedido médio de fractals (800×800×500).
 */
private static final long ESTIMATED_WORK_THRESHOLD = 1_000_000_000L;
```

O `checkAndScale()` passou a calcular:

```java
long totalEstimatedWork = soma de w.getEstimatedWork() para todos os workers
long avgEstimatedWork = totalEstimatedWork / numWorkers

// Scale-up:   avgEstimatedWork > 1_000_000_000
// Scale-down: avgEstimatedWork < 250_000_000
```

O log do AutoScaler agora mostra:
```
[AutoScaler] Workers=3, TotalActive=5, TotalEstWork=240000000000, AvgEstWork=80000000000
```

O `scaleDown()` também passou a usar `getEstimatedWork()` para escolher o worker a terminar.

## Impacto

| Cenário | Antes | Agora |
|---|---|---|
| 1 pedido `2000×2000×2000` (~80B BBs) | **NÃO** escalava (active=1 < 1.0) | **ESCALA** (avgEstWork=80B > 1B) |
| 20 pedidos `100×100×50` (~10M BBs cada) | **ESCALAVA** (active=20 > 1.0) | **NÃO** escala (avgEstWork=200M < 1B) |
| Worker com 1 pedido pesado vs 3 leves | LB escolhia o de 1 (menos active) | LB escolhe o de 3 leves (menos estWork) |
| Scale-down escolhe worker | Podia matar worker com pedido pesado | Mata o worker com menos `estimatedWork` |

### Calibração do threshold

O valor `1_000_000_000L` (1 bilião de BBs) foi escolhido porque:
- Um fractal médio (`800×800×500`) tem custo heurístico de `800×800×500×10 = 3.2B` → dispara scale-up
- Um fractal leve (`200×200×100`) tem custo de `200×200×100×10 = 40M` → NÃO dispara
- O scale-down (threshold/4 = 250M) só acontece quando o sistema está praticamente ocioso

O valor pode ser ajustado via system property `-Dcnv.estwork.threshold=N` se necessário.
