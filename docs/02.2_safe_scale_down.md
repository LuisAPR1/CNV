# Decisão AS — Scale-down seguro com adiamento por drenagem incompleta

> **Data:** 2026-05-21
> **Estado:** ✅ Implementado
> **Ficheiros tocados:** `AutoScaler.java`

---

## Para que serve este documento

Documenta o bug subtil que existia no scale-down e a correcção aplicada. Importante para o relatório (mostra que pensámos em correctness, não só em scaling).

---

## O bug original

A versão anterior do `AutoScaler.scaleDown()` fazia:

```
1. Selecionar worker com menos load (entre os geridos pelo AS)
2. Removê-lo do pool (pára de receber pedidos novos)
3. Esperar até 30s para drenar pedidos em curso
4. terminateInstances(target.instanceId)   ← incondicional
5. lastScalingAction = now
```

O passo 4 corre **mesmo que ainda haja `activeRequests > 0`** após os 30s. Resultado:
- A EC2 é morta.
- Os pedidos em curso são interrompidos a meio.
- Os clientes recebem `connection reset` ou 502.

Em demos com pedidos curtos isto raramente aparece (drenam em <30s). Em produção com pedidos pesados (Fractals 2000×2000×2000 demora alguns minutos), é uma fonte real de perda de pedidos.

---

## A correcção: adiar em vez de matar

```
1. Selecionar worker com menos load (entre os geridos pelo AS)
2. Removê-lo do pool (pára de receber pedidos novos)
3. Esperar até DRAIN_POLL_ITERATIONS × DRAIN_POLL_INTERVAL_MS (15 × 2s = 30s)
4. SE activeRequests > 0:
     - Log "scale-down ADIADO"
     - Re-adicionar worker ao pool
     - NÃO actualizar lastScalingAction
     - return (próximo ciclo do AutoScaler tentará de novo)
   SENÃO:
     - terminateInstances(target.instanceId)
     - lastScalingAction = now
```

Pontos-chave:

### Por que **re-adicionar** ao pool?

Se ficasse fora do pool, o LB nunca mais lhe enviaria pedidos. Mas os pedidos em curso lá dentro já estavam — vão acabar entre os tries do scale-down. Quando o AS retomar a tentativa no próximo ciclo, o worker estaria efectivamente "perdido" (fora do pool, sem novos pedidos a chegar mas com pedidos antigos a terminar lentamente).

Re-adicionar significa:
- O LB pode usá-lo se faltar capacidade noutros workers.
- Os pedidos em curso terminam normalmente.
- O próximo ciclo do AutoScaler vê-o como qualquer outro worker (e pode tentar de novo se a carga continuar baixa).

### Por que **NÃO actualizar** `lastScalingAction`?

`lastScalingAction` controla o cooldown de 60s entre scaling actions. Se actualizássemos, teríamos de esperar 60s antes da próxima tentativa, mesmo que o worker entretanto esvazie. Não actualizar significa:
- O cooldown só conta a partir do **último scale REAL** (terminação concluída ou novo worker).
- Esta tentativa adiada não bloqueia retries imediatos.
- O próximo ciclo (5s depois) re-tenta se a carga ainda for baixa.

### Por que **30s de drenagem** e não mais?

Trade-off:
- Mais tempo: mais tolerante a pedidos longos, mas atrasa a libertação de recursos.
- Menos tempo: liberta mais cedo, mas adia mais frequentemente (mais ciclos para terminar).

30s é um meio-termo razoável para os workloads deste projeto:
- Fractals médio (800×800×500): ~5-10s.
- Fractals pesado (2000×2000×2000): vários minutos — vai exigir múltiplos ciclos para terminar, mas isso é correcto.
- GrayScott / DNA: tipicamente <15s.

## Health-check eviction continua agressivo (e está correcto)

Há outra cadeia de terminação no `AutoScaler.handleUnhealthyEviction()` (`@c:\Users\Luisr\Desktop\IST\CNV\WORK\CNV\loadbalancer\src\main\java\pt\ulisboa\tecnico\cnv\loadbalancer\AutoScaler.java:88-102`): quando o `WorkerPool` falha 3 health-checks consecutivos a um worker, chama este callback que termina a EC2.

Aí **não há drenagem** — e está correcto: o worker já não responde a HTTP, os pedidos em curso já estão a falhar de qualquer forma. Matar a EC2 é o caminho certo (liberta recursos AWS).

A diferença entre os dois fluxos:

| Cenário | Drenagem | Razão |
|---|---|---|
| Scale-down voluntário (carga baixa) | ✅ Sim | Worker está OK, queremos terminar limpo |
| Health-check eviction (worker rebentou) | ❌ Não | Worker já está inutilizável, drenar é fútil |

## Combinação com o packing (02.1)

A política de packing tende a deixar workers idle aparecerem mais rapidamente. Combinada com o scale-down seguro:

1. **Steady state baixo**: poucos pedidos chegam → packing concentra em 1 worker → outros 2 ficam a 0 → AS faz scale-down de 2 em 2 ciclos sem perder nada.
2. **Steady state alto**: muitos pedidos → packing satura → AS dispara scale-up.
3. **Transição alta→baixa**: o último pedido pesado a terminar atrasa scale-down (adiamento), mas eventualmente o cluster consolida.

É a combinação certa para minimizar custo sem perder qualidade de serviço.

## Como testar

Cenário de adiamento:

```bash
# Arrancar 2 workers
java -javaagent:... 8001 &
java -javaagent:... 8002 &

# Arrancar LB com AutoScaler em local mode
java -cp ... LoadBalancer 8080 localhost:8001 localhost:8002

# Disparar 1 pedido MUITO pesado para encher um worker
curl "http://localhost:8080/fractals?w=4000&h=4000&iterations=2000" &

# Logo a seguir, parar de disparar pedidos
# Esperar ~3min (cooldown + drenagem)
```

Esperado: o worker ocupado **não** é morto enquanto o pedido pesado corre. Quando termina, o próximo ciclo do AS faz scale-down clean.
